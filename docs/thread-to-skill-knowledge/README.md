## 1. 背景与痛点

日常使用 AI 工具做复杂任务时，经常会出现这些问题：

- 同类问题需要反复纠正，例如文档写法、代码审查重点、测试边界、技术方案完整性。
- 高价值反馈散落在不同 thread 里，过一段时间后很难复用。
- AI 工具当次被纠正后能做对，但下一次新 thread 可能又回到旧习惯。
- 直接把一次 thread 生成 skill 容易太早固化，规则还没验证充分。
- 团队协作时，每个人和 AI 工具对齐出的经验无法共享，导致重复踩坑。
- 如果没有去噪和领域归类，聊天记录会越来越多，但真正可执行的知识并没有增加。

这套机制把“纠偏过程”拆成脱敏来源记录、AI 初标与人工确认、领域沉淀和 skill 生产 4 个环节，让一次协作产生的经验可以进入长期知识循环。

## 2. 目标

把日常和 AI 工具沟通过程中已经对齐、验证、修正过的经验沉淀下来，形成一套可持续积累的知识库。后续这些知识可以继续整理成不同领域的 skill，例如代码 CR、QA、技术方案、埋点审查、文档写作等。

这套机制的核心不是保存聊天记录，而是把聊天中的有效反馈转成结构化经验。它采用“AI 自动初标 + 人工审核关键项”的方式，从真实协作过程里提取偏好、规则、反例、判断标准和可复用流程，让后续 AI 工具在同类任务里能更早进入正确方向。

本质思路是把日常和 AI 工具沟通、纠偏、对齐出来的聊天内容作为原始数据源。`thread-to-skill` 先自动去除过程噪音，识别并标注其中有质量的问题点、有效回答、用户纠正、反例和最终口径；再按照领域把这些材料整合成稳定知识；最后基于高质量、可复用的知识一键生成或更新 skill。这里的“生成 skill”不是训练模型参数，而是把真实协作中验证过的高质量数据提炼成触发场景、执行规则、工作流程和检查清单，让 skill 来自真实任务，而不是凭空编写。

整体链路可以理解为：

```text
日常 AI 对齐聊天 -> 沉淀为原始数据源（会脱敏） -> AI 自动去噪与标注 -> 领域知识整合 -> 一键生成或更新 skill
```

一句话概括：

```text
把一次次和 AI 工具对齐出来的经验，沉淀成可追溯、可复用、可升级为 skill 的知识资产，并一键制作成 skill。
```

## 3. 适用场景与不适用场景

适用场景：

- 一个 thread 中多次纠正了 AI 工具的思考方式、输出格式或判断标准。
- 某类任务有明显重复性，例如代码 CR、QA、技术方案、埋点审查、文档写作。
- 用户已经确认了一批稳定规则，希望下次不要重复解释。
- 团队希望把个人和 AI 协作经验变成共享知识库。
- 想从多个真实任务中提炼高质量 skill，而不是凭空设计 skill。

不适用场景：

- 一次性问题，没有复用价值。
- 纯事实查询或简单命令执行。
- 规则仍高度依赖某个临时项目背景，无法泛化。
- thread 中包含大量不能脱敏的敏感信息，且没有私有知识库承载。
- 用户还没有确认最终口径，只是临时探索多个可能方向。

## 4. 整体分层

建议把沉淀内容分成 4 层：

| 层级 | 作用 | 产物 |
|------|------|------|
| 来源层 | 保存脱敏后的必要上下文、证据摘要和来源说明 | `source-records/<source-id>/README.md`、`evidence-summary.md`、`redaction-notes.md` |
| 复盘记录层 | AI 自动去噪和初标，人只确认低置信度、冲突和高影响内容 | `review-records/<review-id>/README.md` |
| 领域层 | 按主题沉淀稳定规则和方法论 | `domains/<domain>/README.md` 和子主题文档 |
| Skill 层 | 把成熟领域方法升级成可复用的 AI 能力包 | `skills/<skill-name>/SKILL.md` 和可选资源文件 |

推荐目录结构：

```text
thread-to-skill-knowledge/
  source-records/
    2026-04-30-single-order-tracking/
      README.md
      evidence-summary.md
      redaction-notes.md
      attachments/
  review-records/
    2026-04-30-single-order-tracking-review/
      README.md
      extracted-rules.md
      high-confidence.md
      needs-confirmation.md
      conflicts.md
  domains/
    docs-writing/
      README.md
      final-state-docs.md
    code-review/
      README.md
    qa/
      README.md
    tech-solution/
      README.md
    tracking-spec-audit/
      README.md
      event-naming.md
      error-analysis.md
    ai-collaboration/
      README.md
  skill-candidates/
    deep-spec-audit/
      README.md
      draft-SKILL.md
    code-review-rubric/
      README.md
  skills/
    deep-spec-audit/
      SKILL.md
      agents/
        openai.yaml
      scripts/
      references/
      assets/
```

生成位置：

- 默认在对应仓库根目录下创建 `thread-to-skill-knowledge/` 目录。目录名与本文标题保持关联，同时和 `thread-to-skill` 这个 skill 名区分开。
- 知识库位置可以由用户指定。指定一次后，可以把该位置记录到 `thread-to-skill` skill 文档里，后续执行时按 skill 中记录的位置写入，不再要求用户重复指定。
- 如果没有用户指定位置，也没有在 skill 中记录过位置，则使用默认位置 `<project-root>/thread-to-skill-knowledge/`。
- 记录到 skill 里的路径建议使用相对路径，方便仓库迁移和开源；只有本地私有场景才使用绝对路径。
- 指定路径表示知识库根目录，`source-records/`、`review-records/`、`domains/`、`skill-candidates/`、`skills/` 都直接放在这个目录下。
- `thread-to-skill` 是负责复盘、去噪和沉淀的辅助 skill；`thread-to-skill-knowledge/` 是该 skill 生成和维护的知识库目录。
- 如果是独立知识库仓库，`thread-to-skill-knowledge/` 放在该仓库根目录。
- 如果是业务项目仓库，`thread-to-skill-knowledge/` 也放在项目根目录，不要混进业务项目的 `docs/reports/`、源码目录或测试目录。
- 本文档可以作为方案说明放在 `docs/` 或 `docs/reports/` 中，但后续生成的沉淀材料应进入仓库根目录下的 `thread-to-skill-knowledge/`。

仓库根目录结构：

```text
<project-root>/
  thread-to-skill-knowledge/
    source-records/
    review-records/
    domains/
    skill-candidates/
    skills/
```

开源时建议采用“双仓库”策略：

- 私有仓库保存真实使用过程中的 `source-records/` 和 `review-records/`，并按需做短期留存。
- 开源仓库只发布脱敏后的 `domains/`、`skill-candidates/`、`skills/`，以及少量脱敏示例。
- 如果要公开 `source-records/`，只放示例化的脱敏来源记录；真实项目来源记录默认保留在私有仓库。

统一目录原则：

- 每个可独立管理的对象都使用一个文件夹，例如一个 thread、一次复盘、一个领域、一个候选 skill、一个正式 AI 能力包。
- 每个文件夹用 `README.md` 作为入口说明，负责描述该对象是什么、当前状态、主要结论和相关引用。
- 内容较多时再拆分子文档，例如 `evidence-summary.md`、`extracted-rules.md`、`event-naming.md`、`draft-SKILL.md`。
- 正式 AI 能力包采用通用 Agent Skills 结构：`SKILL.md` 是必需入口，`scripts/`、`references/`、`assets/` 是可选资源目录；Codex 可以额外保留 `agents/openai.yaml` 作为展示层元数据。

目录说明：

| 目录 | 状态 | 用途 | 内容特点 |
|------|------|------|----------|
| `source-records/` | 脱敏来源材料 | 每个来源一个文件夹，保存必要上下文、证据摘要和脱敏说明 | 默认不保存完整对话；`README.md` 写基本信息，`evidence-summary.md` 写证据摘要，`redaction-notes.md` 记录脱敏和未保存内容 |
| `review-records/` | AI 初标与人工确认记录 | 每次复盘一个文件夹，保存 AI 自动初标结果、待确认内容和冲突内容 | AI 先完成去噪、分类和置信度判断，人只处理低置信度、冲突和高影响内容 |
| `domains/` | 领域知识库 | 按主题沉淀稳定规则和方法论 | 每个领域一个文件夹，`README.md` 作为入口；只保留最终态规则，不保留中间方案 |
| `skill-candidates/` | 候选草稿区 | 每个候选 skill 一个文件夹，存放准备升级成 skill 的方法论草稿 | 可以比正式 skill 更完整，允许保留待验证内容、反例和补充说明 |
| `skills/` | 正式可用区 | 每个正式 AI 能力包一个文件夹，至少包含 `SKILL.md` | 内容要短、稳、可执行；Codex 和 Claude Code 默认复用同一个 `SKILL.md` |

`skill-candidates/` 和 `skills/` 的区别：

- `skill-candidates/` 是候选区。某个领域的方法论已经有价值，但还没有足够稳定，或者还需要继续补充触发场景、反例、检查清单和验证方式时，先放在这里。
- `skills/` 是正式区。只有当候选内容已经稳定、可复用、可执行，并且能被目标 AI 工具在后续任务中直接使用时，才升级为正式 AI 能力包。
- 候选草稿可以写得更长，方便积累材料；正式 AI 能力包要压缩成执行规则，避免把历史过程、一次性背景和长篇复盘塞进去。
- `skill-candidates/<skill-name>/README.md` 负责记录候选 skill 的目标、适用场景、成熟度和待验证问题；`draft-SKILL.md` 可以承载拟升级为正式 AI 能力包的草稿。
- `skills/<skill-name>/SKILL.md` 是正式 AI 能力包的通用入口；Codex 和 Claude Code 默认复用同一个文件。正式发布时只保留运行所需内容，其他长篇材料继续留在候选区或领域目录中。

升级链路：

```text
thread 复盘 -> domains 领域目录 -> skill-candidates 候选草稿 -> skills 正式 AI 能力包
```

## 5. 案例

以一次复杂埋点审查 thread 为例：

| 阶段 | 产物 | 内容 |
|------|------|------|
| 来源层 | `source-records/2026-04-30-single-order-tracking/evidence-summary.md` | 保存脱敏后的任务背景、关键证据、用户修正摘要和最终结论 |
| 复盘记录层 | `review-records/2026-04-30-single-order-tracking-review/README.md` | 提取用户偏好、已确认规则、反例、正确做法和待确认问题 |
| 领域层 | `domains/tracking-spec-audit/README.md` | 沉淀埋点审查的通用流程、检查清单、错误分析维度、事件命名规则 |
| 候选 skill | `skill-candidates/deep-spec-audit/draft-SKILL.md` | 把领域方法整理成候选 skill，继续验证触发场景和检查清单 |
| 正式 AI 能力包 | `skills/deep-spec-audit/SKILL.md` | 发布稳定、简洁、可执行的深度规格审查能力 |

这个链路的重点是逐层提纯：

```text
脱敏来源摘要 -> 有价值反馈 -> 稳定领域规则 -> 候选 skill -> 正式 AI 能力包
```

## 6. 来源层：保存脱敏来源材料

每个有价值的 thread 先保存一份脱敏来源记录。来源层的目标不是保存完整聊天原文，而是为后续规则提供可追溯、可公开控制的证据摘要。

默认策略：

- 不默认保存完整 thread 原文。
- 保存脱敏后的任务背景、关键决策、用户修正摘要、最终结论和必要证据片段。
- 对未保存或已脱敏的信息做说明，方便后续理解证据缺口。
- 如需短期保存完整原文，只放在私有本地或受控私有仓库，并设置清理周期。
- 开源仓库只保留脱敏后的来源摘要，不保留完整聊天记录。

建议目录命名：

```text
YYYY-MM-DD-主题
```

示例：

```text
source-records/
  2026-04-30-single-order-tracking/
    README.md
    evidence-summary.md
    redaction-notes.md
  2026-05-02-code-review-preference/
    README.md
    evidence-summary.md
    redaction-notes.md
  2026-05-08-tech-solution-writing/
    README.md
    evidence-summary.md
    redaction-notes.md
```

来源层文件说明：

| 文件 | 作用 |
|------|------|
| `README.md` | 来源基本信息、主题、相关领域、关联复盘和可公开级别 |
| `evidence-summary.md` | 脱敏后的关键上下文、用户确认、反例、最终结论和必要证据 |
| `redaction-notes.md` | 记录哪些内容被脱敏、未保存或仅保留在私有环境 |
| `attachments/` | 只保存脱敏后的截图、日志、片段或其他附件 |

`redaction-notes.md` 不保存敏感内容本身，只保存脱敏处理说明。它用于说明 `evidence-summary.md` 不是完整原文，哪些信息被替换、模糊化、删除或没有保存。

示例：

```md
# 脱敏说明

## 已脱敏内容

| 类型 | 处理方式 | 原因 |
|------|----------|------|
| 订单号 | 替换为 `<order_id>` | 用户数据 |
| 手机号 | 替换为 `<phone>` | 隐私信息 |
| 内部接口名 | 替换为 `<internal_api>` | 公司内部信息 |
| 本地路径 | 替换为 `<repo_path>` | 环境信息 |

## 未保存内容

| 内容类型 | 原因 |
|----------|------|
| 完整聊天原文 | 可能暴露用户表达习惯和敏感上下文 |
| 未脱敏截图 | 包含业务数据和内部 UI |
```

文件职责可以简单理解为：

```text
evidence-summary.md = 保存脱敏后的证据摘要
redaction-notes.md = 说明哪些内容被脱敏或没有保存
```

## 7. 复盘记录层：AI 自动初标与人工确认

thread 结束后，优先由 AI 做自动初标。AI 根据上下文语义判断哪些内容已经被用户确认、哪些是过程噪音、哪些是反例、哪些可能需要继续确认。人工不需要逐条阅读完整 thread，只需要审核 AI 标出的低置信度、冲突和高影响内容。

建议按以下类型抽取：

| 类型 | 说明 |
|------|------|
| 用户明确偏好 | 用户明确表达过“以后要这样做”的习惯或标准 |
| 已确认规则 | 经过多轮确认后稳定下来的规则 |
| 反例 / 踩坑 | AI 工具做错过、用户指出过的问题 |
| 正确做法 | 修正后的最终做法 |
| 判断标准 | 以后遇到类似问题时如何判断 |
| 可复用流程 | 可以迁移到其他任务的步骤 |
| 检查清单 | 下次任务完成前需要逐项检查的内容 |
| 适用场景 | 什么时候使用这条规则 |
| 不适用场景 | 什么时候不要套用这条规则 |
| 来源记录 | 方便追溯到脱敏来源摘要 |

AI 自动初标的分层：

| 分层 | 处理方式 | 典型信号 |
|------|----------|----------|
| 高置信度 | AI 可直接整理成待合并规则 | 用户明确说“好的”“就这样”“以这个为主”“补到文档里”“删掉” |
| 中置信度 | AI 放入待确认区，给出判断依据 | 用户多次纠正后形成偏好，但没有明确说可通用化 |
| 低置信度 | AI 提出问题，等待人工确认 | 语义不完整、上下文不足、可能只是当前项目特例 |
| 冲突项 | AI 单独列出冲突双方和建议处理方式 | 新规则与已有领域规则不一致，或同一 thread 前后口径不一致 |
| 高影响项 | AI 即使有较高把握，也应提示人工确认 | 会影响正式 AI 能力包、团队规范、开源内容或安全脱敏 |

AI 自动初标时重点识别这些信号：

- 用户明确确认：例如“好的”“就这样”“按这个来”“以新版为主”。
- 用户明确否定：例如“不要这样写”“这个去掉”“没必要”“你理解错了”。
- 用户纠正 AI：通常对应反例、偏好或方法论。
- 用户要求补文档：通常说明该规则已经进入最终方案。
- 多次重复出现：说明可能是长期偏好。
- 最终文档状态：已落地内容优先作为最终态规则来源。

人工主要确认这些内容：

- 是否要升级成通用规则。
- 是否只是当前项目特例。
- 是否和已有领域规则冲突。
- 是否包含敏感信息。
- 是否值得进入候选 skill。
- 是否会误导后续任务。

推荐复盘记录目录：


| 文件 | 作用 |

## 8. 领域层：按主题沉淀稳定方法

领域目录中的文档是长期维护的知识资产，不是一轮 thread 的复盘。它们应该只保留已经稳定、可复用、跨任务有效的内容。

为了便于开源和长期扩展，`domains/` 下统一采用“每个领域一个文件夹”的结构。每个领域文件夹必须有一个 `README.md` 作为入口文档；当领域内容变多时，再按子主题拆分更多 md 文件。

推荐先建立这些领域：

| 领域目录 | 入口文档 | 覆盖内容 |
|----------|----------|----------|
| `docs-writing/` | `docs-writing/README.md` | 最终态文档、标题编号、措辞偏好、避免保留过程方案 |
| `tracking-spec-audit/` | `tracking-spec-audit/README.md` | 埋点、数仓、trace、漏斗、错误分析、新旧实现对齐 |
| `code-review/` | `code-review/README.md` | 代码 CR 的关注顺序、风险判断、输出结构 |
| `qa/` | `qa/README.md` | 测试设计、验收口径、边界场景、回归风险 |
| `tech-solution/` | `tech-solution/README.md` | 技术方案结构、方案完整性、实现细节、风险和验收 |
| `ai-collaboration/` | `ai-collaboration/README.md` | 用户对 AI 工具思考深度、输出方式、复盘方式的偏好 |

`README.md` 负责写领域总览、核心原则、通用流程和总检查清单；子主题文档负责承载更细的规则、例子、反例和专题检查清单。每次 thread 复盘后，只把稳定结论合并到对应领域目录，不把完整 thread 内容直接塞进领域入口文档。

## 9. Skill 层：从领域目录升级成 AI 能力包

不要一个 thread 结束就立刻生成 skill。更好的方式是先沉淀到领域目录里，等规则重复出现、足够稳定后再升级。

适合升级成 skill 的判断标准：

- 同类规则至少在多个任务或多个 thread 中反复出现。
- 不是一次性项目决策，而是可复用方法。
- 有明确触发场景。
- 有稳定工作流程。
- 有可执行检查清单。
- 有正例和反例。
- 能帮助 AI 工具下次少走弯路。

正式 AI 能力包的内容应该保持短、稳、可执行，重点写：

- 什么时候使用。
- 核心原则。
- 工作流程。
- 检查清单。
- 输出要求。
- 验证方式。

不建议把完整 thread、长篇复盘、一次性背景全部塞进正式 AI 能力包。脱敏来源记录和领域目录继续保留，正式 AI 能力包只承载执行时真正需要的规则。

## 10. thread-to-skill 的使用方式与推荐流程

`thread-to-skill` 是用于维护这套知识库的辅助 skill。它负责把一次或多次 thread 中的有效对齐内容去噪、分类、沉淀到领域目录，并判断是否适合升级成候选 skill 或正式 AI 能力包。

### 10.1 安装与更新

`thread-to-skill` 已通过 `@simplexd/simple-skills` 发布到 npm。`simple-skills` 是一个 AI skill 合集安装器，默认会安装包内 `skills/` 目录下的所有 skill；随着仓库后续加入更多能力，同一个安装命令会一起分发这些 skill。

交互式安装：

```bash
npx @simplexd/simple-skills@latest
```

安装器会提示选择 AI runtime 和安装位置。当前支持 Codex、Claude Code、Cursor、GitHub Copilot、generic agents、Gemini CLI 和 Qwen Code。

Codex 全局安装：

```bash
npx @simplexd/simple-skills@latest --codex --global
```

只安装 `thread-to-skill`：

```bash
npx @simplexd/simple-skills@latest --codex --global --skill thread-to-skill
```

更新已安装的 `thread-to-skill`：

```bash
npx @simplexd/simple-skills@latest --codex --global --skill thread-to-skill --force
```

查看当前包内包含哪些 skill：

```bash
npx @simplexd/simple-skills@latest --list
```

安装或更新后，重启对应 AI 工具，让新 skill 生效。

这个安装方式适合个人和团队共享：团队可以把真实协作中反复验证过的规则沉淀到本仓库的 `skills/` 目录，再通过同一个 npm 包分发给不同 AI 工具，避免每个新 thread 都重新解释偏好、流程和反例。

常用触发方式：

```text
请使用 thread-to-skill 复盘这个 thread，把有价值的对齐内容沉淀下来。
```

更明确的触发方式：

```text
请使用 thread-to-skill：
1. 复盘当前 thread
2. 去掉过程噪音
3. 提取我的偏好、已确认规则、反例、正确做法和方法论
4. 判断应该合并到哪些领域目录
5. 判断是否适合升级成 skill
6. 输出中文 md
```

处理已保存 thread 文档：

```text
请使用 thread-to-skill 处理 source-records/<source-id>/evidence-summary.md，
生成复盘记录到 `review-records/`，并判断应该合并到哪些 domains。
```

直接沉淀到知识库：

```text
请使用 thread-to-skill，把当前 thread 沉淀到仓库根目录下的 thread-to-skill-knowledge 知识库里。
目录结构按本文件的规范来。
```

从领域目录生成候选 skill：

```text
请使用 thread-to-skill，基于 domains/<domain>/README.md 和相关子主题文档，
生成一个 <skill-name> 的候选 skill。
```

后续只要在请求里明确提到 `thread-to-skill`，AI 工具就会按该 skill 的流程执行。

推荐流程：

每次重要 thread 收尾时，可以让 AI 工具执行以下流程：

1. 保存脱敏来源记录到 `source-records/`。
2. 由 AI 自动生成 thread 初标结果，放到 `review-records/`。
3. AI 按高置信度、待确认、冲突项和高影响项分层。
4. 人工只确认待确认、冲突和高影响内容。
5. 将高置信度且无冲突的稳定规则合并到领域目录或子主题文档。
6. 合并领域目录时，只保留最终态规则，不保留中间讨论过程。
7. 判断是否已经成熟到进入候选 skill 或正式 AI 能力包。

可以直接使用这个提示词：

```text
请复盘这个 thread：
1. 去掉过程噪音
2. 提取我明确表达过的偏好、规则、反例和方法论
3. 按高置信度、待确认、冲突项和高影响项分层
4. 按领域归类
5. 判断哪些应该进入已有领域目录或子主题文档
6. 判断哪些已经成熟到可以升级成 skill
7. 输出最终态内容，不要保留讨论过程
```

合并领域目录时可以使用：

```text
把这次复盘结果合并到 thread-to-skill-knowledge/domains/<domain>/README.md 或对应子主题文档。
如果已有规则冲突，先列出来让我确认。
文档只保留最终态规则，不要写过程方案。
```

生成 AI 能力包时可以使用：

```text
基于 thread-to-skill-knowledge/domains/<domain>/README.md 和相关子主题文档，生成一个 <skill-name> AI 能力包。
要求：
1. skill 内容简洁、可执行
2. 包含触发场景、核心原则、工作流程、检查清单和验证方式
3. 不要把历史 thread 原文塞进 skill
4. 使用中文描述
```

## 11. 角色分工

团队协作时，建议把职责拆开：

| 角色 | 主要职责 |
|------|----------|
| Thread 参与者 | 在日常协作中指出问题、确认规则、补充反例和正确做法 |
| 标注者 | thread 结束后做去噪复盘，提取稳定规则和待确认问题 |
| 领域维护者 | 维护某个 `domains/<domain>/` 目录，合并稳定规则，处理冲突 |
| Skill 维护者 | 将成熟领域方法整理为候选 skill 或正式 AI 能力包，并负责校验 |
| 审核者 | 对开源内容做脱敏检查、质量检查和最终发布确认 |

小团队或个人使用时，一个人可以同时承担多个角色。角色拆分的目的不是增加流程，而是确保知识沉淀有明确责任边界。

## 12. 质量标准

一条可以合并进领域目录的规则，至少应该满足：

- 能跨多个任务复用，而不是只服务一次性项目。
- 已经被用户确认，或有明确证据支撑。
- 表达的是最终态规则，不保留中间过程。
- 能指导 AI 工具下一次做出更好的判断或输出。
- 有清晰适用场景，必要时也说明不适用场景。
- 如果来自具体项目，已经抽象成通用说法。

一条暂时不应该合并的内容包括：

- 只是临时探索，还没有确认。
- 和现有规则冲突但尚未解决。
- 过于依赖某个业务、接口、项目路径或内部背景。
- 只是聊天里的情绪表达，没有可执行规则。
- 已经被后续讨论推翻。

正式 AI 能力包的质量标准：

- 触发场景明确。
- 流程短而可执行。
- 检查清单能直接指导任务完成。
- 不包含长篇 thread 原文或敏感互动细节。
- 不包含废弃方案。
- 通过 skill 校验。

## 13. 冲突处理机制

当新复盘内容和已有领域规则冲突时，不直接覆盖。建议按以下方式处理：

1. 在复盘文档里记录冲突点。
2. 标明冲突双方的来源。
3. 说明可能的原因，例如适用场景不同、规则过期、项目差异、用户偏好变化。
4. 给出建议处理方式。
5. 等用户或领域维护者确认后，再更新领域目录。

冲突记录模板：

```md
## 冲突记录

| 冲突点 | 旧规则 | 新规则 | 来源 | 建议处理 |
|--------|--------|--------|------|----------|
| ... | ... | ... | ... | ... |
```

确认后的领域文档只保留最终规则；冲突过程可以留在 `review-records/<review-id>/README.md` 中用于追溯。

## 14. 开源脱敏规则

如果这套知识库要开源，需要区分私有材料和公开材料。

建议策略：

- `source-records/` 默认私有；开源仓库只放脱敏后的示例来源记录，完整 thread 原文不进入开源仓库。
- `review-records/` 默认私有；开源前需要去掉待确认、冲突过程和敏感上下文，只保留可公开的示例。
- `domains/` 可以开源，但要移除公司、项目、接口、用户数据和内部路径。
- `skill-candidates/` 可以开源脱敏版本。
- `skills/` 可以开源，但内容应尽量通用。

开源前检查：

- 是否包含真实用户数据、订单号、手机号、地址、邮箱、账号等。
- 是否包含公司内部项目名、接口名、仓库路径、业务策略或未公开信息。
- 是否包含截图、日志、代码片段中的敏感内容。
- 是否把一次性项目规则误写成通用规则。
- 是否可以把具体业务名替换成通用描述。

推荐做法：

- 私有环境可以短期保存完整 thread，用于近期复盘，过期后删除。
- 开源仓库主要放领域规则、候选 skill、正式 AI 能力包和少量脱敏示例。
- 来源记录只保留摘要或私有引用，不公开完整 thread。

## 15. 成功指标

这套机制是否有效，可以从这些指标观察：

| 指标 | 说明 |
|------|------|
| 重复纠错次数下降 | 同类问题不需要反复提醒 AI 工具 |
| 领域文档复用次数上升 | 新任务开始时能直接引用已有领域规则 |
| 候选 skill 转正式 AI 能力包的数量 | 说明沉淀内容逐渐成熟 |
| 正式 AI 能力包使用后返工减少 | AI 能力包能让 AI 工具更接近用户预期 |
| 新成员理解成本下降 | 团队成员能通过领域目录快速理解协作标准 |
| 冲突规则闭环率 | 冲突能被记录、确认并合并成最终规则 |

## 16. 维护规则

为了让知识库长期可用，建议遵守这些规则：

- 来源层默认只保存脱敏摘要和必要证据，不保存完整 thread 原文。
- 复盘记录层优先由 AI 自动初标，人工只确认待确认、冲突和高影响内容。
- 领域入口文档和子主题文档只写最终态规则。
- Skill 只写执行时需要的最小知识。
- 冲突规则不要直接覆盖，先标记冲突并让用户确认。
- 已经废弃的方案不要保留在领域目录和 skill 中。
- 每条重要规则尽量能追溯到来源 thread。
- 定期清理 `review-records/`，把有价值内容合并到领域目录。

## 17. FAQ

### 17.1. 为什么不直接把每个 thread 都生成 skill？

单个 thread 里的规则可能还不稳定，直接生成 skill 容易把临时结论固化。更好的方式是先进入领域目录，经过多次复用和验证后再升级。

### 17.2. 为什么还需要来源层？

来源层用于保留可追溯依据，但不默认保存完整聊天原文。后续如果某条规则有争议，可以回到脱敏后的来源摘要、关键证据和脱敏说明中确认上下文。完整 thread 只适合在私有环境中短期保存或加密保存。

### 17.3. 领域目录和 skill 有什么区别？

领域目录是知识库，可以更完整地沉淀规则、例子、反例和来源。skill 是执行入口，要短、稳、可操作，只保留 AI 工具执行任务时真正需要的内容。

### 17.4. 候选 skill 有什么价值？

候选 skill 是缓冲区。它允许先整理草稿、验证触发场景和检查清单，再发布正式 AI 能力包，避免过早固化。

### 17.5. 这套机制是否绑定某个 AI 工具？

不是。只要是和 AI 协作中产生的有效反馈，都可以按这套方式沉淀。Codex skill、Claude Project、Cursor Rules、团队 Prompt 规范等都可以作为落地形态。

## 18. 这套机制的价值

这套机制可以把日常协作中的“纠偏成本”转化为长期资产。

短期价值：

- AI 工具更快理解用户偏好。
- 减少重复解释。
- 让复杂任务的复盘更系统。
- 让文档和方案更接近用户想要的最终形态。

长期价值：

- 形成个人或团队的 AI 协作知识库。
- 为不同领域的 skill 提供稳定素材。
- 让 skill 来自真实任务，而不是凭空设计。
- 把人工反馈沉淀成可迁移、可复用、可验证的方法论。
